# v5 — Run B prime regime, NO aux loss

## Hypothesis
Reverting to Run B prime's adapter shape (LoRA r=16 / α=8, lm_expert q,v only, pure LoRA, uniform action MSE) on the new dataset (512² + repositioned camera + stretched scripted phases + tightened xy_tolerance + per-episode color injection) is the single change that returns us to the only checkpoint that ever cleared ≥ 2/10 SR. v5 = test that hypothesis.

## Pre-launch decisions (2026-04-27)

- **Dataset target reduced**: 752 demos in the bd issue → **157 real demos** (50 + 7 + 50 + 50 from the 4-shard parallel gen) due to disk math at 512² + long-phase per-demo footprint. v3 had 370 demos at 256²; v5 has 157 demos at 512² + 2× longer phases, so total visual-token volume is comparable but episode count is lower. See `reports/2026-04-27_8rf_parallel_demo_orchestrator.md` for the disk math.
- **Aux cube-localization loss DEFERRED to v5b**: vla_kitting-mu7 returned PARTIAL (R²_x=0.852 PASS, R²_y=0.473 FAIL, cos-sim=0.658 PASS). The bd issue's protocol maps PARTIAL → AUX_0.05× action weight, but implementing the head + plumbing through lerobot's modeling_smolvla.py requires ~2–3h that would push training start past safe morning landing. v5b filed as `vla_kitting-3xp` (P2) to be triggered iff v5 SR ∈ [3/30, 10/30] per 8ux's "directional progress" branch.
- **Step budget**: bd issue says "~100k". Realistic at v5 batch=16 + 512² images and ~0.7–1.0 s/step is ~25–40k steps overnight. Aiming for "as many as fits" with intermediate evals every 10 epochs (= 10k steps). The headline acceptance is the SR number on configs/eval_seed_30.json at the latest available checkpoint.
- **One stuck shard**: shard 1 hung mid-demo-8 phase 2 around 03:09 UTC — process at 96% CPU but no log/HDF5 writes for 80 minutes. SIGTERM ignored; SIGKILL succeeded. 7 demos preserved. Other shards continued. Root cause not investigated; suspected Isaac Sim internal stall under multi-process contention. If reproducible, file as a follow-up.
